# Plan: Public API Documentation Improvements

## Goal

Improve the Weather API's OpenAPI documentation with XML doc comments, request/response examples, and a new metadata endpoint, making the Scalar API reference (`/scalar/v1`) comprehensive and self-explanatory for consumers.

## Current State

- **API definition** (`apps/weather-api/Program.cs`): Minimal API endpoints with `.WithName()` for each route. No `.WithDescription()`, `.WithSummary()`, `.Produces()`, or `.WithOpenApi()` calls. The OpenAPI spec is auto-generated by `builder.Services.AddOpenApi()` and served at `/openapi/v1.json` via `app.MapOpenApi()`.
- **Scalar UI** (`apps/weather-api/Program.cs`): `app.MapScalarApiReference()` serves the interactive docs at `/scalar/v1`. Currently shows all endpoints but with minimal descriptions — just route paths and auto-inferred parameter types.
- **Project file** (`apps/weather-api/WeatherApi.csproj`): Targets `net9.0`. Has `Microsoft.AspNetCore.OpenApi` version `9.0.11` and `Scalar.AspNetCore` version `2.13.12`. No `<GenerateDocumentationFile>` property is set, so XML comments are not included in the build output.
- **Models** (`apps/weather-api/Models/`): `WeatherForecast.cs` has `[Required]` and `[NotMapped]` attributes but no XML doc comments. `Minion.cs` has `[Required]`, `[MaxLength]`, and `[JsonConverter]` attributes but no XML comments.
- **Traefik** (`traefik/traefik-dynamic.yml`): The Scalar UI is accessible directly at `http://localhost:5221/scalar/v1` (not proxied through Traefik). The admin-app links to it at this URL. No public-facing API docs route through Traefik exists.

## Implementation Steps

### 1. Enable XML documentation generation

Edit `apps/weather-api/WeatherApi.csproj`:

```xml
<PropertyGroup>
  <TargetFramework>net9.0</TargetFramework>
  <Nullable>enable</Nullable>
  <ImplicitUsings>enable</ImplicitUsings>
  <GenerateDocumentationFile>true</GenerateDocumentationFile>
  <NoWarn>$(NoWarn);1591</NoWarn>
</PropertyGroup>
```

The `1591` suppression prevents build warnings for public types without XML comments (since not all framework types will have them).

### 2. Add XML doc comments to models

Edit `apps/weather-api/Models/WeatherForecast.cs`:

```csharp
/// <summary>
/// A weather forecast record with temperature and conditions for a specific date.
/// </summary>
public class WeatherForecast
{
    /// <summary>Unique identifier for the forecast.</summary>
    public int Id { get; set; }

    /// <summary>The date of the forecast (YYYY-MM-DD).</summary>
    /// <example>2026-03-28</example>
    [Required]
    public DateOnly Date { get; set; }

    /// <summary>Temperature in degrees Celsius.</summary>
    /// <example>22</example>
    public int TemperatureC { get; set; }

    /// <summary>Weather condition label (e.g., Mild, Warm, Scorching).</summary>
    /// <example>Mild</example>
    public string? Summary { get; set; }

    /// <summary>Temperature in degrees Fahrenheit (computed from TemperatureC).</summary>
    [NotMapped]
    public int TemperatureF => 32 + (int)(TemperatureC / 0.5556);
}
```

Edit `apps/weather-api/Models/Minion.cs` similarly with XML comments for each property.

### 3. Add endpoint metadata to Program.cs

Edit `apps/weather-api/Program.cs` to enhance each endpoint with OpenAPI metadata. For example:

```csharp
forecasts.MapGet("/", async (IWeatherForecastRepository repo) =>
    Results.Ok(await repo.GetAllAsync()))
    .WithName("GetWeatherForecasts")
    .WithSummary("List all forecasts")
    .WithDescription("Returns all weather forecasts in the database, ordered by ID. No pagination — use the /history endpoint for filtered, paginated results.")
    .Produces<IEnumerable<WeatherForecast>>(StatusCodes.Status200OK);

forecasts.MapGet("/{id:int}", async (int id, IWeatherForecastRepository repo) =>
{
    var result = await repo.GetByIdAsync(id);
    return result is null ? Results.NotFound() : Results.Ok(result);
})
.WithName("GetWeatherForecastById")
.WithSummary("Get a forecast by ID")
.WithDescription("Returns a single weather forecast by its unique integer ID.")
.Produces<WeatherForecast>(StatusCodes.Status200OK)
.Produces(StatusCodes.Status404NotFound);

forecasts.MapPost("/", async (WeatherForecast input, IWeatherForecastRepository repo) =>
{
    try
    {
        var created = await repo.CreateAsync(input);
        return Results.Created($"/weatherforecast/{created.Id}", created);
    }
    catch (NotSupportedException ex)
    {
        return Results.Problem(ex.Message, statusCode: StatusCodes.Status405MethodNotAllowed);
    }
})
.WithName("CreateWeatherForecast")
.WithSummary("Create a new forecast")
.WithDescription("Creates a new weather forecast. The TemperatureF field is computed automatically and should not be provided in the request body.")
.Produces<WeatherForecast>(StatusCodes.Status201Created)
.Produces(StatusCodes.Status405MethodNotAllowed);
```

Apply the same pattern to `PUT /{id}`, `DELETE /{id}`, and all minion endpoints.

### 4. Add the `/weatherforecast/metadata` endpoint

Add to `apps/weather-api/Program.cs`:

```csharp
forecasts.MapGet("/metadata", () =>
{
    var metadata = new
    {
        validSummaries = new[]
        {
            new { label = "Freezing", tempRange = "below -5°C" },
            new { label = "Bracing", tempRange = "-5°C to 0°C" },
            new { label = "Chilly", tempRange = "0°C to 5°C" },
            new { label = "Cool", tempRange = "5°C to 12°C" },
            new { label = "Mild", tempRange = "12°C to 18°C" },
            new { label = "Warm", tempRange = "18°C to 24°C" },
            new { label = "Balmy", tempRange = "24°C to 30°C" },
            new { label = "Hot", tempRange = "30°C to 35°C" },
            new { label = "Sweltering", tempRange = "35°C to 40°C" },
            new { label = "Scorching", tempRange = "above 40°C" },
        },
        temperatureRange = new { minC = -60, maxC = 60 },
        apiVersion = "1.0",
        documentation = "/scalar/v1",
    };
    return Results.Ok(metadata);
})
.WithName("GetWeatherMetadata")
.WithSummary("API metadata and reference data")
.WithDescription(
    "Returns reference data for the Weather API: valid summary labels with their " +
    "temperature ranges, the accepted temperature range, and the API version. " +
    "Useful for building client UIs with dropdowns and validation.");
```

The label-to-temperature mapping mirrors the `LABEL_RANGES` dictionary in `dag_quality_report.py` (lines 69-80), ensuring consistency between the API documentation and the quality evaluation logic.

### 5. Configure OpenAPI document metadata

Edit `apps/weather-api/Program.cs`, enhance the `AddOpenApi` call:

```csharp
builder.Services.AddOpenApi(options =>
{
    options.AddDocumentTransformer((document, context, ct) =>
    {
        document.Info.Title = "Weather Forecast API";
        document.Info.Version = "1.0";
        document.Info.Description =
            "REST API for managing weather forecasts. Supports CRUD operations on forecasts " +
            "and automated forecast generation via minions. Data is stored in PostgreSQL and " +
            "changes are streamed via Debezium CDC to Kafka.";
        document.Info.Contact = new()
        {
            Name = "Weather Platform Team",
            Url = new Uri("https://github.com/joebarbere/claude-hello-world"),
        };
        return Task.CompletedTask;
    });
});
```

### 6. Add request/response examples via OpenAPI transformers

.NET 9 Minimal APIs support `WithOpenApi()` for per-endpoint schema customization. Add examples to key endpoints:

```csharp
forecasts.MapPost("/", ...)
    .WithOpenApi(operation =>
    {
        operation.RequestBody.Content["application/json"].Example =
            new Microsoft.OpenApi.Any.OpenApiObject
            {
                ["date"] = new Microsoft.OpenApi.Any.OpenApiString("2026-03-28"),
                ["temperatureC"] = new Microsoft.OpenApi.Any.OpenApiInteger(22),
                ["summary"] = new Microsoft.OpenApi.Any.OpenApiString("Mild"),
            };
        return operation;
    });
```

### 7. Optionally expose Scalar through Traefik

To make API docs accessible at `https://localhost:8443/api-docs` (consistent with the rest of the platform), add a route in `traefik/traefik-dynamic.yml`:

```yaml
api-docs-router:
  rule: "PathPrefix(`/api-docs`)"
  entryPoints:
    - websecure
  service: weather-api
  priority: 20
  middlewares:
    - api-docs-rewrite
  tls: {}
```

```yaml
api-docs-rewrite:
  replacePathRegex:
    regex: "^/api-docs(.*)"
    replacement: "/scalar/v1$1"
```

This is optional — the direct URL already works for local development. The Traefik route is useful if the platform is ever deployed to a shared environment.

## Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| `WithOpenApi()` lambda examples may break if the OpenApi library version changes | Pin the `Microsoft.AspNetCore.OpenApi` version in the csproj. The example API is stable in .NET 9. |
| XML documentation file increases build output size | Negligible — the XML file is typically < 20KB. It is not included in the container image unless explicitly copied. |
| The metadata endpoint hardcodes label ranges — drift risk with `dag_quality_report.py` | Add a comment in both files referencing each other. Long-term, extract the label ranges to a shared configuration file or database table. |
| Traefik API docs route may conflict with future paths starting with `/api-docs` | Low risk. The path is distinctive and unlikely to conflict. |

## Dependencies

- **No hard dependencies**. This is a standalone improvement to the existing API.
- **Benefits from**: Any new endpoint added by other plans (history, alerts, quality) should follow the documentation patterns established here (`.WithSummary()`, `.WithDescription()`, `.Produces()`).
- **Benefits**: "Admin-app: quality score panel" — the metadata endpoint helps client UIs build dropdowns and validation rules.

## Estimated Complexity

**Small** — All changes are additive metadata on existing code. No new database tables, no new Angular components, no infrastructure changes. The bulk of the work is writing descriptive text for each endpoint and model property.
