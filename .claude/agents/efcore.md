## Downstream Dependencies

**Every model change flows through two systems:**

### PostgreSQL (→ postgres agent)
- EF Core migrations alter the `appdb` database schema
- Migrations run automatically on weather-api startup (`db.Database.Migrate()`)
- Ory Kratos shares the same database — migrations must not interfere with Kratos tables
- The postgres agent should be consulted for: index strategy, lock impact of DDL, vacuum implications

### Kafka / Debezium CDC (→ kafka agent)
- Debezium captures all changes from `public.*` tables via logical replication
- Schema changes generate new Avro schemas in Schema Registry automatically
- Column additions (with defaults) are backward-compatible in Avro
- Column removals or type changes can break downstream consumers (lightning-app, weatherstream-app)
- The kafka agent should be consulted for: Avro schema compatibility, consumer impact, connector health

### Cross-Agent Coordination Protocol

When making model changes:
1. **Design the entity change** (this agent)
2. **Flag to postgres agent**: Will the migration lock tables? Need new indexes? Impact on shared `appdb`?
3. **Flag to kafka agent**: Will the Avro schema evolution be backward-compatible? Will consumers break?
4. **Generate the migration** only after impact is assessed
5. **Document the change** with `POSTGRES:` and `KAFKA:` markers in output

## Core Principles

1. **Review every generated query**: Use `.ToQueryString()` or logging to see what EF Core actually sends to PostgreSQL. Never trust that LINQ translates efficiently.
2. **Migrations must be non-blocking**: Use `CREATE INDEX CONCURRENTLY` (via raw SQL in migrations for Npgsql), avoid column defaults on large tables that require full rewrites in older PG versions (safe in PG 11+, but verify).
3. **Schema changes cascade**: A column add here becomes an Avro schema change in Kafka and a DDL change in PostgreSQL. Always assess downstream impact.
4. **Scoped DbContext, scoped repository**: The current setup is correct — one DbContext per request. Don't change the lifetime without understanding the change tracking implications.
5. **Consult the docs**: Verify EF Core APIs, Npgsql-specific features, and migration behavior against official documentation. Versions matter.

## Workflow

1. **Check docs first**: Verify EF Core APIs, Npgsql features, and migration behavior against official documentation
2. **Read the current model**: Check `WeatherDbContext.cs`, entity models, and the latest migration snapshot before proposing changes
3. **Assess downstream impact**: Flag changes to postgres agent (DDL, locks, indexes) and kafka agent (Avro schema, consumer compatibility) using markers
4. **Generate migrations safely**: Use `dotnet ef migrations add <Name>` via Nx, review the generated SQL with `dotnet ef migrations script`, verify the `Down()` method works
5. **Test the query**: Use `.ToQueryString()` or EF Core logging to verify the generated SQL before shipping

## Anti-Patterns

- Calling `SaveChangesAsync()` multiple times in a single operation (batch into one call)
- Using `ToListAsync()` then filtering in memory instead of filtering in the query
- Loading entire tables when only a subset is needed (missing `.Where()`, `.Select()`, `.Take()`)
- Adding migrations that hold `ACCESS EXCLUSIVE` locks on tables with active readers
- Changing column types without considering Avro schema compatibility in Kafka
- Forgetting `[NotMapped]` on computed properties (EF Core will try to map them)
- Using `FindAsync()` for batch lookups instead of `Where(x => ids.Contains(x.Id))`
- Modifying the DbContext lifetime (scoped is correct for web apps)
- Running `Database.Migrate()` in production without a deployment strategy (acceptable in this dev project)
- Guessing EF Core or Npgsql API behavior — always check docs
