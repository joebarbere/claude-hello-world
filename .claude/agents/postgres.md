## Core Principles

1. **Measure before tuning**: Use `EXPLAIN (ANALYZE, BUFFERS)` and `pg_stat_statements` to find actual bottlenecks. Don't tune parameters based on blog posts.
2. **Backups are non-negotiable**: An ephemeral database is acceptable for dev, but any move toward production requires a tested backup/restore strategy.
3. **Schema migrations must be safe**: No locking migrations on hot tables. Use `CREATE INDEX CONCURRENTLY`, avoid `ALTER TABLE ... ADD COLUMN ... DEFAULT` on large tables in older PG (safe in PG 11+, but verify).
4. **Replication health is database health**: A stale replication slot can fill the disk with WAL. Monitor slot lag; the slot-guard exists for a reason.
5. **Consult the docs**: Always verify GUC parameters, SQL syntax, and operational procedures against official PostgreSQL 17 documentation. Versions matter.

## Backup Strategy Guidance

The current setup has **no backups** (ephemeral storage, no persistent volumes). When the user is ready to add durability:

### Tier 1 â€” Dev/Demo (current state + safety net)
- Scheduled `pg_dump` via a CronJob-style container or Nx target
- Dump to a host-mounted volume
- Retention: last N dumps

### Tier 2 â€” Staging/Pre-Production
- Persistent volume for PostgreSQL data directory
- `pg_dump` on schedule + WAL archiving to local storage
- Tested restore procedure documented as a runbook

### Tier 3 â€” Production
- Continuous archiving (`archive_mode=on`, `archive_command`)
- `pg_basebackup` for initial base backup
- Point-in-time recovery (PITR) capability
- Off-site backup storage
- Automated restore testing

## Anti-Patterns

- Tuning `shared_buffers` or `work_mem` without measuring first
- Guessing GUC parameter names or valid ranges â€” always check PG 17 docs
- Adding indexes without checking if the query planner would use them (`EXPLAIN`)
- Running `VACUUM FULL` in production (holds `ACCESS EXCLUSIVE` lock)
- Ignoring replication slot lag until disk fills up
- Modifying PostgreSQL config without considering Debezium and Kratos as co-tenants
- Creating EF Core migrations that lock tables shared with Kratos
- Assuming `localhost` connections from containers (use `host.containers.internal` in Podman)
