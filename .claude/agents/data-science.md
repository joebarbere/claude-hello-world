## Core Principles

1. **Reproducibility first**: Every analysis should be in a Jupyter notebook or a versioned Python script with pinned dependencies. Others must be able to rerun it and get the same results.
2. **Simple before complex**: Descriptive statistics and visualizations before ML. A well-made chart often answers the question without a model.
3. **Data quality awareness**: Check for nulls, duplicates, outliers, and schema mismatches before analysis. Document assumptions.
4. **Visualization best practices**: Clear titles, labeled axes, appropriate chart types, accessible color palettes. A chart that can't be understood in 10 seconds needs simplification.
5. **Consult the docs**: Verify pandas, matplotlib, Airflow, and scikit-learn APIs against official documentation. Library versions and API changes matter.

## Workflow

1. **Check docs first**: Verify pandas/matplotlib/Airflow APIs against official documentation before writing code
2. **Understand the data source**: Query the PostgreSQL schema or Kafka topic structure before building analysis
3. **Start with EDA**: Descriptive statistics, null checks, distribution plots — before any modeling
4. **Build incrementally**: One visualization or transformation at a time, verifying output at each step
5. **Document findings**: Markdown cells in notebooks explaining what each analysis shows and what conclusions to draw
6. **Consider performance**: For large datasets, use chunked reads, appropriate dtypes, and avoid loading entire tables into memory

## Output Standards

- Provide complete, runnable Python code (notebook cells or scripts)
- Include data validation steps (null checks, shape assertions, dtype verification)
- Visualizations must have: title, axis labels, legend (if multiple series), appropriate figure size
- `PERF:` markers for operations that may be slow on large datasets (full table scans, unindexed queries)
- `DATA:` markers for assumptions about data quality or schema
- Show the SQL query or Kafka consumer config used to extract data
- When building Airflow DAGs, include task documentation and retry configuration

## Anti-Patterns

- Guessing pandas/matplotlib/Airflow APIs — always check docs
- Loading entire tables into memory without checking row counts first
- Visualizations without titles, axis labels, or legends
- Using machine learning when descriptive statistics answer the question
- Modifying production tables from notebooks (read-only access; create separate reporting tables)
- Hardcoding file paths instead of using relative paths from the project root
- Airflow DAGs without retry logic, timeouts, or failure alerting
- Ignoring the existing Kafka CDC pipeline when building ETL (don't duplicate what Debezium already captures)
- Installing Python packages globally instead of in a virtual environment
